# Ubuntu Delivery Preparation Design

## Goal

在现有 Ubuntu 首次部署手册的基础上，补齐交付清单、现场操作准备、回滚说明、验收记录模板，并明确本次涉及文档统一使用 UTF-8 编码，方便交付人员在现场按一份文档完成部署与交接。

## Scope

- 继续使用 `docs/2026-04-09-ubuntu-deployment-adaptation.md` 作为唯一主手册。
- 在主手册中新增交付包清单、部署前登记项、现场操作顺序、回滚说明、验收记录模板、最终交付检查表。
- README 保持最小入口调整，必要时把入口描述改成“部署与交付准备文档”。
- 本次修改涉及的 Markdown 文档按 UTF-8 保存，避免出现编码不一致导致的乱码。

## Approaches

### Option 1: 单文档整合

把首次部署步骤和交付准备全部整合进现有 Ubuntu 手册。

Pros:
- 交付人员只看一份文档即可完成准备、部署和交接。
- 最适合现场操作，减少来回跳转。

Cons:
- 文档会变长，需要控制结构清晰度。

### Option 2: 主手册 + 模板附录

保留主手册，新增独立交付清单或验收模板文档。

Pros:
- 文档职责更清晰。

Cons:
- 交付现场需要在两份文档间切换。

### Option 3: 文末追加最小补丁

仅在现有 Ubuntu 手册末尾追加交付准备条目。

Pros:
- 改动最小。

Cons:
- 结构不够稳定，后续继续维护会变乱。

## Recommendation

采用 Option 1。对交付人员来说，单文档整合最稳，且现有 Ubuntu 手册已经承担主入口角色，继续扩展它比再拆文档更合适。

2. 部署前信息登记表

4. 回滚说明
5. 验收记录模板
6. 最终交付检查表

README 仅做以下最小变化：

- 保留 Ubuntu 文档入口
- 如有必要，将入口文案调整为“Ubuntu 部署与交付准备文档”

## UTF-8 Requirement

- 本次修改涉及的 Markdown 文件统一以 UTF-8 保存。
- 以文件实际读取内容为准判断是否存在乱码，不以 PowerShell 控制台显示结果单独判断文件损坏。
- 不做无意义的大范围重写，只修正本次变更涉及文件的编码一致性。

## Validation

- 重新读取目标文档，确认中文内容正常。
- 保持现有文档相关测试通过。
- 在提交前确认 `git diff` 只包含预期的文档内容和编码相关变化。
