# Ubuntu Delivery Doc Design

## Goal

补齐一份面向运维/交付人员的 Ubuntu 首次部署文档，使其可以在不了解项目内部实现的前提下，按步骤完成服务器准备、应用部署、服务启动与首次验收。

## Scope

- 新增独立交付文档，聚焦 Ubuntu 22.04/24.04 首次部署。
- 文档覆盖环境准备、代码落盘、Python 依赖、`.env` 配置、数据库初始化、`systemd`、Caddy、端口放行、验收与排障。
- README 仅增加最小入口链接，不重复展开整套交付步骤。

## Approach Options

### Option 1: 独立交付手册

新增一份单独的 Ubuntu 部署交付文档，按执行顺序组织内容。

Pros:
- 最贴合交付场景，读者不需要在 README 中筛信息。
- 可以加入预期结果、检查命令、失败排查，降低执行门槛。

Cons:
- 与 README 中现有 Linux 部署章节存在少量信息重复。

### Option 2: 扩写 README

直接把 README 的 Linux 部署章节扩成完整操作手册。

Pros:
- 入口统一。

Cons:
- README 会显著变长，不利于后续维护。
- 首页文档与交付手册职责混杂。

### Option 3: README 保持简版 + 新手册跳转

README 保留简版说明，同时新增独立交付手册并添加入口。

Pros:
- 兼顾首页概览与交付落地。
- 变更最小且职责清晰。

Cons:
- 需要维护两处入口文字。

## Recommendation

采用 Option 3。正式操作步骤写入独立文档，README 仅保留摘要和跳转入口，避免首页过重，同时让交付人员拿到清晰的执行手册。

## Document Structure

正式交付文档按以下章节组织：

1. 适用范围与部署目标
2. 服务器前置条件
3. 目录准备与项目上传
4. Python 虚拟环境与依赖安装
5. `.env` 配置项说明
6. 数据库初始化与预检查
7. `systemd` 服务安装与开机启动
8. Caddy 反向代理配置
9. 防火墙与安全组放行
10. 首次部署验收清单
11. 常见故障排查

## Validation

- 文档中每个关键步骤都尽量提供命令和预期结果。
- 仅引用仓库中已经存在的脚本和配置文件，不引入新的部署机制。
- README 链接应指向新文档，便于交付人员从首页进入。
